# 3-Day Implementation Plan

## 목표

3일 안에 박람회 현장에서 사용할 수 있는 최소 기능 웨이팅 서비스를 완성한다.

초기 버전은 다음 범위까지만 포함한다.

- QR 진입용 사용자 웨이팅 등록 화면
- 휴대폰 번호 입력 및 개인정보 동의
- 사용자 남은 순서 조회 화면
- 사용자 웨이팅 취소
- 관리자 웨이팅 목록
- 관리자 현장도착 확인/입장완료/노쇼 처리
- 관리자 호출 기반 SOLAPI 카카오 알림톡 발송
- 로컬 embedded H2 실행 환경
- 운영 PostgreSQL + Docker Compose 실행 환경

초기 버전에서는 다음은 제외한다.

- WebSocket/SSE 실시간 갱신
- Redis
- Kafka 또는 메시지 큐
- 별도 프론트엔드 애플리케이션
- 복잡한 관리자 권한 관리
- 다중 행사/다중 부스 관리

## 구현 원칙

- Spring Boot 단일 애플리케이션으로 구현한다.
- Thymeleaf 서버 렌더링을 사용한다.
- 로컬 DB는 embedded H2를 사용하고, 운영 DB는 PostgreSQL을 사용한다.
- 대기 순서는 별도 컬럼으로 저장하지 않고 조회 시 계산한다.
- 카카오 알림톡은 관리자의 개별 호출 또는 부족 인원 호출 액션에서 발송한다.
- 알림톡 중복 발송은 `notified_at` 값으로 막는다.
- 3일 안에 운영 가능한 수준을 목표로 하고, 확장 기능은 뒤로 미룬다.

## Day 1: 프로젝트 기반과 도메인 완성

### 목표

애플리케이션이 PostgreSQL과 연결되고, 웨이팅 데이터를 저장/조회/상태 변경할 수 있는 백엔드 기반을 만든다.

### 작업

1. 의존성 정리
   - Spring JDBC 추가
   - PostgreSQL JDBC Driver 추가
   - 로컬/테스트용 H2 추가
   - 초기 스키마는 `schema.sql`로 관리

2. Docker Compose 작성
   - 운영용 `app`, `db`
   - PostgreSQL volume
   - PostgreSQL 환경 변수 자리 확보

3. 애플리케이션 설정
   - `application.yaml`에 datasource 설정
   - `.env` 기반 환경 변수 연동
   - local profile 필요 여부 판단

4. 도메인 모델 구현
   - `WaitingEntry`
   - `WaitingStatus`
   - 생성/호출/현장도착/입장완료/노쇼/취소 시간 필드

5. Repository 구현
   - 진행 중 웨이팅 중복 등록 확인
   - 관리자 목록 조회
   - 입장 완료 목록 조회
   - 상태 변경 처리

6. Service 구현
   - 웨이팅 등록
   - 남은 순서 조회
   - 웨이팅 취소
   - 현장도착 확인
   - 입장완료
   - 노쇼 처리
   - 관리자 호출 액션에서 알림톡 발송 대상자 계산 진입점

7. 기본 테스트 작성
   - 웨이팅 등록
   - 중복 등록 방지
   - 남은 순서 계산
   - 취소 후 순서 변경
   - 입장완료 후 순서 변경
   - 노쇼 처리 후 순서 변경

### 완료 기준

- `./gradlew test`가 통과한다.
- PostgreSQL 컨테이너가 정상 실행된다.
- 웨이팅 등록 데이터가 DB에 저장된다.
- 상태 변경 후 남은 순서가 다시 계산된다.

## Day 2: 화면과 카카오 알림톡 연동

### 목표

사용자와 관리자가 브라우저에서 실제 운영 흐름을 수행할 수 있게 만들고, SOLAPI 카카오 알림톡 발송을 연결한다.

### 작업

1. 사용자 화면 구현
   - `GET /`
   - 휴대폰 번호 입력
   - 개인정보 동의 체크박스
   - 등록 버튼
   - 모바일 기준 레이아웃

2. 웨이팅 등록 처리
   - `POST /waitings`
   - 휴대폰 번호 정규표현식 검증
   - 휴대폰 번호 자동 하이픈 입력 보조
   - 중복 등록 시 안내 메시지 표시
   - 등록 성공 시 `/waitings/{id}`로 redirect

3. 사용자 대기 상태 화면
   - `GET /waitings/{id}`
   - 현재 상태 표시
   - 남은 순서 표시
   - 예상 대기시간은 1팀당 3분으로 표시
   - 새로고침 버튼
   - 웨이팅 취소 버튼
   - 현장도착/입장완료/노쇼 상태 표시
   - 취소 완료 상태 표시

4. 사용자 취소 처리
   - `POST /waitings/{id}/cancel`
   - `WAITING` 상태만 취소 가능
   - 취소 성공 시 `status = CANCELED`, `canceled_at = now()`
   - 취소된 사용자는 남은 순서와 관리자 기본 목록에서 제외
   - 취소 후 사용자 상태 화면으로 redirect

5. 관리자 화면 구현
   - `GET /admin/waitings`
   - 기본 화면에는 `WAITING`, `CALLED`, `ARRIVED` 상태만 등록 순서로 표시
   - 기본 화면 요약에는 `WAITING`, `CALLED`, `ARRIVED` 상태만 포함
   - `CANCELED`, `ENTERED`, `NO_SHOWED` 상태는 기본 화면 요약과 상세 목록에서 제외
   - 웨이팅 목록이 10명을 초과하면 페이징 처리
   - 휴대폰 번호 표시
   - 현재 상태
   - 남은 순서
   - 현장도착 확인 버튼
   - 입장완료 버튼
   - 노쇼 버튼
   - 취소처리 버튼(`WAITING` 상태만)
   - `GET /admin/waitings/entered`
   - 입장 완료된 사용자는 별도 화면에 등록 순서 또는 입장완료 시각 순서로 표시
   - 입장 완료 목록이 10명을 초과하면 페이징 처리

6. 관리자 처리 구현
   - `POST /admin/waitings/{id}/arrive`
   - `POST /admin/waitings/{id}/enter`
   - `POST /admin/waitings/{id}/no-show`
   - `POST /admin/waitings/{id}/cancel`
   - 처리 후 관리자 목록으로 redirect

7. SOLAPI 카카오 알림톡 연동
   - SOLAPI 설정 properties 구성
   - 알림톡 sender wrapper 구현
   - 운영용 카카오 알림톡 발송 구현
   - test용 no-op sender 구현
   - 발송 성공 시 `status = CALLED`, `notified_at = now()`
   - 발송 실패 시 로그 기록, `notified_at` 미변경

8. 알림톡 발송 정책 테스트
   - 부족 인원 호출 시 현장 관리 인원 5명까지 발송 대상 계산
   - `notified_at`이 있으면 재발송하지 않음
   - 발송 성공 시 `CALLED`로 변경
   - 발송 실패 시 `WAITING` 유지

### 완료 기준

- 모바일 브라우저에서 사용자가 웨이팅 등록을 완료할 수 있다.
- 사용자가 새로고침으로 남은 순서를 확인할 수 있다.
- 사용자가 본인의 웨이팅을 취소할 수 있다.
- 관리자가 현장도착 확인/입장완료/노쇼 처리를 할 수 있다.
- 관리자가 호출한 사용자에게만 카카오 알림톡 발송 로직이 실행된다.
- SOLAPI 키가 없어도 알림톡 호출 기능을 제외한 local 개발과 test 실행이 가능하다.

## Day 3: 운영 준비와 현장 검증

### 목표

박람회 현장에서 바로 사용할 수 있도록 배포, 설정, 장애 대응, 수동 검증을 마무리한다.

### 작업

1. 관리자 접근 보호
   - 현재 초기 구현에는 포함하지 않는다.
   - 운영 노출 전 최소한의 Basic Auth 또는 단일 관리자 비밀번호 적용을 검토한다.

2. 운영 설정 정리
   - `.env.example` 작성
   - DB 접속 정보 정리
   - SOLAPI 환경 변수 정리
   - 발신번호 설정 확인 항목 문서화

3. 실행 검증
   - `./gradlew bootRun`
   - 운영 compose 배포 검증
   - 애플리케이션 기동 확인
   - DB volume 유지 확인
   - 재시작 후 데이터 유지 확인

4. QR 운영 준비
   - 사용자 등록 URL 확정
   - QR 코드 생성 대상 URL 정리
   - 모바일 화면에서 입력/등록/조회 확인
   - 모바일 화면에서 취소 확인

5. 현장 운영 시나리오 테스트
   - 대기자 10명 등록
   - 대기자 11명 이상 등록 후 관리자 기본 목록 페이징 확인
   - 현장 관리 인원 5명 기준 부족 인원 호출 대상 확인
   - 입장완료 후 부족 인원 호출 대상 확인
   - 입장완료된 사용자가 기본 목록에서 빠지고 입장 완료 목록에 표시되는지 확인
   - 노쇼 처리 후 부족 인원 호출 대상 확인
   - 사용자 취소 후 부족 인원 호출 대상 확인
   - 취소된 사용자가 기본 목록 요약과 상세 목록에 표시되지 않는지 확인
   - 중복 등록 방지 확인
   - 이미 발송된 사용자 중복 발송 방지 확인

6. 로그와 에러 처리 확인
   - 카카오 알림톡 발송 실패 로그 확인
   - 잘못된 휴대폰 번호 입력 처리
   - 존재하지 않는 웨이팅 ID 접근 처리
   - 이미 처리된 사용자 재처리 방지
   - 이미 취소된 사용자 재취소 방지

7. 최종 문서 정리
   - 실행 방법
   - 운영 환경 변수
   - 관리자 URL
   - QR URL
   - 장애 시 확인할 로그

### 완료 기준

- Docker Compose로 앱과 PostgreSQL이 함께 실행된다.
- 실제 또는 테스트 SOLAPI 설정으로 카카오 알림톡 발송 흐름을 확인했다.
- 재시작 후 웨이팅 데이터가 유지된다.
- 모바일 사용자 화면과 관리자 화면에서 전체 시나리오가 동작한다.
- 사용자가 모바일 화면에서 웨이팅을 취소할 수 있다.
- 관리자 기본 화면은 처리 대상자만 보여주고, 입장 완료 목록은 별도 화면에서 확인할 수 있다.
- 관리자 기본 목록과 입장 완료 목록은 10명 초과 시 페이징된다.
- 현장 운영자가 현장도착 확인/입장완료/노쇼 처리를 수행할 수 있다.

### 반드시 구현

- 웨이팅 등록
- 개인정보 동의 체크
- 남은 순서 조회
- 웨이팅 취소
- 관리자 목록
- 현장도착 확인

### 시간이 부족하면 단순화

- 관리자 인증은 운영 노출 전 별도 적용
- 화면 디자인은 최소 모바일 사용성만 확보
- 카카오 알림톡 발송 이력 테이블은 만들지 않고 `notified_at`과 로그로 처리
- Flyway가 부담되면 JPA ddl-auto 또는 간단한 schema.sql로 시작

### 3일 안에 하지 않음

- 실시간 순서 갱신
- 관리자 권한 세분화
- 부스 여러 개 지원
- 통계 대시보드
- 카카오 알림톡 재시도 배치
- 클라우드 오토스케일링

## 리스크와 대응

| 리스크 | 대응 |
| --- | --- |
| SOLAPI 발신번호 등록 또는 API 설정 지연 | 테스트에서는 no-op sender로 검증하고, 실제 발송은 승인된 SOLAPI 설정으로 확인 |
| 현장 인터넷 불안정 | 카카오 알림톡 발송 실패 시 상태를 `WAITING`으로 유지하고 로그로 확인 |
| 중복 알림톡 발송 | `notified_at is null` 조건과 트랜잭션 처리로 방지 |
| 잘못된 관리자 클릭 | 이미 `ENTERED`, `NO_SHOWED`, `CANCELED` 상태인 사용자는 재처리 방지 |
| 사용자가 실수로 취소함 | 취소 버튼에 확인 문구를 두고, 취소 후에는 관리자에게 문의하도록 안내 |
| DB 데이터 유실 | PostgreSQL volume 사용, 운영 전 재시작 테스트 |
| 시간이 부족함 | 관리자 인증/화면 디자인/알림톡 이력 테이블을 최소화 |
